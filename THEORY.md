# A theory of `~/.claude`

## What this is

This repository occupies the same directory the Claude Code CLI uses for its working state on this machine. There is no build, no install, no deploy: the files committed here and the files the tool reads at startup are the same files. The system models one specific thing — the user's portable half of his contract with Claude Code. Claude Code owns the other half: the lifecycle events, the directory layout under `~/.claude/`, the skills and plugins it ships with, the auto-memory machinery. This repo holds the small set of files that belong to the user, follow him across machines via `origin/main`, and persist through reinstalls of the CLI. Everything in the repo is in service of keeping the user's side of that line clear, and maintainable from inside a Claude session.

## The organizing ideas

**`.gitignore` is the schema, not a filter.** Every path it lists is owned by the Claude Code runtime, generated by use, and not the user's to edit. Every path it does not list is the user's. Three other files defer to this document: `biome.json` sets `vcs.useIgnoreFile: true`, `.prettierignore` is a symlink to `.gitignore`, and `.claude/CLAUDE.md` narrates the same boundary in prose. One source of truth, three downstream consumers. A file is either tracked-and-formatted or ignored-and-runtime-owned; it cannot be in both modes, and the formatter scope cannot drift away from the version-control scope, because there is nowhere for that drift to hide.

**The repo speaks to Claude.** The two `CLAUDE.md` files are written for a Claude session as their primary reader, not for a human browsing the repo on GitHub. `CLAUDE.md` is loaded by Claude Code in every session and tells Claude who the user is and how to address him: philoserf, Go and TypeScript, terse. `.claude/CLAUDE.md` is loaded only when a session is working inside this directory; it tells Claude what the layout means, which paths are safe to touch, and which vendor-shipped skills to route changes through. The split is by reading occasion: the first file is read everywhere, the second only here. `README.md` is the one document addressed to a human, and its primary function is to deflect — its first heading after the title is "Don't use this directly."

**One knob, one explanation.** `settings.json` contains exactly one key, `skipDangerousModePermissionPrompt: true`. The "Current settings.json" section of `.claude/CLAUDE.md` records why that key is there. The pattern, not the count, is the invariant: every key is paired with a sentence of rationale in the prose document. The reader of next month should not encounter a setting whose purpose isn't named in the same repo.

**Editing the directory is routed through Claude's own tools.** `.claude/CLAUDE.md` instructs that settings changes go through the `update-config` skill and keybinding changes through `keybindings-help`. These are skills the runtime provides, not local code. The repo's stance is that the proper tool for operating on this directory is Claude Code itself, invoked from inside a session. The maintainer's role is to keep the prose accurate so that Claude, on next reading, still knows what is safe.

## The seams

The internal seam — the line between user-owned and runtime-owned — is made of `.gitignore` entries. Every entry is root-anchored with a leading slash so it binds only at the top of the tree, not wherever a name happens to recur. The runtime-owned set is grouped by purpose: runtime state, credentials, telemetry, machine-local settings, per-project state, editor noise. The grouping is for the human maintainer; the tooling sees a flat list. Credentials get a deliberately broad pattern (`*credentials*.json`) — the only place in the file where breadth is preferred to precision, because the cost of accidentally tracking a credential file outweighs the cost of an over-match.

The external seam is to Claude Code itself, and it is unsupervised. The runtime can rename a directory, add a new top-level path, change which skills it ships, or alter the encoding of `projects/<encoded-cwd>/`. None of these would break the repo loudly; they would silently misrepresent the boundary, and the "Layout" section of `.claude/CLAUDE.md` would become quietly wrong. This is the only place in the system where staying correct depends on noticing something outside the repo.

The per-machine seam is `settings.local.json`. It is gitignored. Any setting the user makes interactively that he does not want to share across machines lives there. The line between tracked `settings.json` and ignored `settings.local.json` is drawn case by case when the user edits, not by any structural rule.

The seam between user data and runtime territory is `projects/<encoded-cwd>/memory/`. Auto-memory files about the user are written there by Claude Code, but the `<encoded-cwd>` is a per-machine hash, so the directory's identity is not portable. `.claude/CLAUDE.md` acknowledges this and points to `git add -f <path>` as the way across when a specific memory deserves to cross. The system does not try to make all memories portable; it makes the boundary crossable on demand.

## What this accommodates

A new fact about the user goes into `CLAUDE.md` as a bullet under "About me" or as a new section. A new setting goes into `settings.json` paired with a line in `.claude/CLAUDE.md` explaining its purpose. A new safety annotation about a runtime path goes into the "Safety rules" section. A new tracked file type means extending `taskfile.yml` with `format:<type>` and `lint:<type>` tasks; the formatter scope follows automatically because both prettier and biome already consult `.gitignore`. Each of these is a single coordinated edit in known places, and each preserves the rule that the boundary documents agree.

## What would require rethinking

Introducing local extensions into the directory — skills, hooks, agents, auto-loaded rule files — would dissolve the cleanest property the system has. Each new component directory would have to be classified (track or ignore), have its file types added to the formatter scope, and acquire an entry in the layout and safety sections of `.claude/CLAUDE.md`. The architecture does not forbid this; it just stops being self-explanatory the moment more than one such directory exists. A maintainer reaching for a local hook to solve a single irritation should first ask whether a plugin or vendor skill already does the work, because the cost of adding one local component is the price of admission to maintaining several.

A change in the upstream Claude Code directory layout would force coordinated edits in `.gitignore` and in `.claude/CLAUDE.md`. These two documents share no syntactic link — keeping them aligned is a human responsibility. If you find the two disagreeing, trust `.gitignore` and update the prose; the runtime-owned set is the load-bearing one.

A move away from `bun` and `bunx` as the runners in `taskfile.yml` would touch every task. The taskfile expresses a preference for `bun` over `npm`/`yarn`/`pnpm`, but only the taskfile records that preference; there is no enforcement and no rule file backing it.

## Uncertainties

Plugins are absent from the tracked tree. `settings.json` enables none. Whether the user runs without plugins, enables them interactively into `settings.local.json` so they are per-machine, or has not yet settled on which to use, cannot be told from the files alone. If a Claude session shows plugin-provided skills active without any diff in tracked `settings.json`, they live per-machine by design.

`.prettierignore` is a symlink to `.gitignore`. On a filesystem or tool that does not follow symlinks, prettier would lose its ignore source. The codebase is implicitly Unix-only on that account. Whether this matters to the user — whether he ever runs the formatters from a non-Unix machine — is not stated.

`.claude/CLAUDE.md` names `update-config` and `keybindings-help` as the proper editing routes, but does not pin them to a marketplace or version. They are presumably vendor-shipped and stable. If a future Claude Code release renames them, the prose becomes stale advice silently — there is no test that would catch it.
